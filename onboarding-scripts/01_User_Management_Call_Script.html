1
Why User Management Comes First

We always start with User Management because every other part of the platform depends on it. The users you create here are the people who will own tickets, log time, get dispatched to jobs, process billing, and run reports. The permissions you assign determine what each of those people can see and do. The groups you create will be referenced directly in System Settings when we configure ticket routing — so getting this right first means everything downstream will work the way you expect.

→ Connects ToUser groups flow directly into System Settings ticket defaults, the Dispatch Board for scheduling, and ticket assignment throughout the platform. Permission sets control access to Billing, Quoting, Customers, Admin, and every other module.
▲ Platform ImpactIf users don't have the right permissions, they'll either be blocked from doing their job or have access to sensitive financial data they shouldn't see. If user groups aren't set up before System Settings, you won't be able to configure your default ticket routing.
2
Creating a User

Go to Admin > User Management > Add User. Let's walk through the key fields together:

  • Email — this is also their login username. The system sends a password reset link here, so it must be their actual work email.
  • Role — a high-level label like Technician, Dispatcher, or Admin. Think of it as a job title in the system, not a security setting.
  • Permission Set — this is the security setting. It controls exactly what the user can access and do. We'll build these next.
  • User Group — how the system knows which team this person belongs to for ticket routing and scheduling.
  • Active toggle — must be on. Deactivating a user immediately removes their access.
❓ PromptLet's create your first user now. Who should we start with?
💡 Internal NotePassword reset links expire in 12 hours. Remind the client to tell new users to check their email right away. If a user misses it, an admin can resend from the User Management list.
3
Permission Sets

Permission sets answer one question: what is each type of user allowed to see and do in Rev.io PSA? Every module — Ticketing, Billing, Quoting, Admin, Inventory — has its own set of permissions you can toggle on or off. The best practice is to build one permission set per role type on your team, then assign users to the set that matches their job.

Go to Permission Sets > Add Permission Set. Name it by role — 'Technician,' 'Dispatcher,' 'Admin.' Then toggle on the access that role needs. You can always come back and adjust these as you learn what your team needs.

❓ PromptWhat roles do you have on your team? Let's build a permission set for your most common role right now.
▲ Platform ImpactPermission sets are how you protect sensitive data. A technician probably shouldn't see billing or be able to delete customer records. An admin needs full access. Getting these right early prevents support headaches later.
→ Connects ToPermissions apply across every module — Billing, Quoting, Customers, Admin settings, Inventory, and Reports. If a user can't see something they need, the first place to check is their permission set.
4
User Groups

User groups are how Rev.io PSA knows which team a ticket or scheduled job should go to. When a ticket comes in — whether from email, the customer portal, or created manually — the system needs to know who should see it. That's controlled by the Default Ticket User Group setting, which we'll set in System Settings in a few minutes. It references the groups you build here.

Go to User Groups > Add User Group, name it, and add users. A user can belong to multiple groups — someone can be on both the general support team and an escalation team at the same time.

❓ PromptHow is your team structured — do you have separate groups for different work types, skill sets, or regions?
→ Connects ToUser groups feed directly into System Settings (default ticket routing), the Dispatch Board (filter by team), and ticket assignment (assign to a group vs. an individual). They also show up in reporting so you can measure performance by team.

Action Items

  • Add all team members as users
  • Build one permission set per role type
  • Create at least one user group before moving to System Settings
  • Remind new users to set their password within 12 hours
